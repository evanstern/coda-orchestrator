# Feature Brief: Inline vs. Branch Decision Framework

## Goal
Add a decision framework that codifies when changes should be made inline in the orchestrator session vs. on a feature branch via `coda feature`.

## Context
During the bootstrapping session, the orchestrator made soul-level changes (SOUL.md, AGENTS.md) inline without branching. The user caught it. This is a drift risk -- the bias toward "it's small, I'll do it here" needs a clear rule.

## Requirements

1. Create a decision framework document or section that defines:
   - **Inline-safe**: memory files, daily logs, learnings, MEMORY.md updates, /reflect proposals, status responses -- anything observational or ephemeral
   - **Branch-required**: SOUL.md changes (even Evolved Identity), AGENTS.md, PROJECT.md, scope.json, designs/, any structural/config change, anything that benefits from PR review
   - **Grey zone guidance**: when unsure, default to branch

2. The framework should be referenced from AGENTS.md so the orchestrator follows it at boot time.

3. Keep it simple -- a short decision table or flowchart, not a policy document.

## Deliverable
- A file (suggest `designs/inline-vs-branch.md` or a new section in AGENTS.md) containing the framework
- Update AGENTS.md to reference it if it's a separate file
- Commit, push, open PR

## Non-goals
- Don't build enforcement tooling (that depends on task spawning)
- Don't modify SOUL.md
